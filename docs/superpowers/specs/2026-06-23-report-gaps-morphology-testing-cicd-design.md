# Закрытие пробелов отчёта: морфология поиска, нагрузочное тестирование, CI

**Дата:** 2026-06-23
**Контекст:** Сверка дипломного отчёта с кодом выявила три заявления, не подкреплённых реализацией. Эта спека описывает их закрытие. Три блока независимы и реализуются отдельно.

## Блок 1. Морфология поиска (русский стеммер)

**Сервис:** `search-service`

**Проблема:** `EventDocument` использует `analyzer = "standard"` на текстовых полях — только токенизация + lowercase, без стемминга. Отчёт §2.4/§484 заявляет морфологический анализ русского языка.

**Решение:** заменить анализатор на встроенный в Elasticsearch `russian` (tokenizer + lowercase + russian_stop + russian_stemmer) на полях:
- `title`, `shortDescription`, `fullDescription` → `@Field(type = FieldType.Text, analyzer = "russian")`
- `location` оставить `standard` (топонимы), `tags` остаётся `Keyword`, остальные поля без изменений.

`russian` — встроенный анализатор ES, дополнительных `@Setting`/JSON не требуется. Match-запрос (`searchByKeyword`) автоматически использует тот же анализатор для разбора запроса, поэтому стемминг работает симметрично (запрос «мероприятия» → стем «мероприят» → находит «мероприятие»).

**Обязательный шаг развёртывания:** смена анализатора меняет маппинг индекса. Существующий индекс `events` на `standard` не перестроится сам. Нужно:
1. удалить индекс `events` (`DELETE /events` в ES);
2. перезапустить `search-service` (Spring Data ES пересоздаст индекс с новым маппингом);
3. вызвать `/api/events/reindex` (наполнить из PostgreSQL — Elasticsearch здесь производная копия).

**Вне scope:** транслитерация «реакт→React» (кириллица↔латиница) — это синонимы/транслит, отдельная задача. Описать в отчёте как ограничение / future work.

**Критерий приёмки (read-back через приложение):**
- до: `GET /api/search/events?query=мероприятия` и `?query=разработке` — не находят / плохо;
- после: находят соответствующие события (стемминг работает);
- регрессия: однословный и многословный латинский поиск («React», «Data Engineering») продолжают работать (HTTP 200).

## Блок 2. Нагрузочное тестирование (k6)

**Артефакт:** `load-test/events-load.js` + `load-test/README.md` (вне Maven).

**Что нагружаем** (ключевые публичные read-эндпоинты через gateway `http://localhost:8180`):
- `GET /api/events` — список;
- `GET /api/events/{id}` — карточка (кешируется Redis);
- `GET /api/search/events?query=...` — поиск.

**Профиль нагрузки:** ступенчатый ramp VU (напр. stages: 30s→10 VU, 1m→50 VU, 1m→100 VU, 30s→0). `thresholds`: `http_req_duration p(95) < 500ms`, `http_req_failed < 1%`.

**Результат:** измеренные p95/RPS. Используются, чтобы заменить «модельные значения» таблицы 1.2 отчёта на реальные.

**Запуск:** `k6 run load-test/events-load.js` по поднятому стеку. В CI не включается (нагрузочные гоняются вручную).

**Критерий приёмки:** скрипт отрабатывает по живому стеку, выводит сводку с p95 и RPS по каждому эндпоинту.

## Блок 3. CI (GitHub Actions)

**Артефакт:** `.github/workflows/ci.yml`. Репозиторий: `NUKI1223/EventHubKzNew`.

**Триггер:** `push` и `pull_request` в `master`.

**Джобы:**
- **backend:** `actions/checkout` → `actions/setup-java` (Temurin 21) + кэш Maven → `mvn -B -ntp verify` в корне (мультимодульная сборка: все сервисы + `eventhubkz-common`, прогон unit и интеграционных тестов на Testcontainers). Docker доступен на `ubuntu-latest`, поэтому Testcontainers-тесты (`*IT.java`, `PostgresTestcontainer`) работают.
- **frontend:** `actions/setup-node` → `npm ci` + `npm run build` в `frontend/`.

**Результат:** рабочий CI, зелёный статус-бейдж в README (подкрепляет §2.7). Бейдж добавить в `README` (если есть) — опционально.

**Не делаем:** сборку/пуш Docker-образов (CD) — без реального целевого окружения смысла нет; описать как возможное расширение.

**Критерий приёмки:** workflow проходит синтаксическую проверку; после пуша на GitHub джобы зелёные (это проверяет пользователь, локально GitHub Actions не запустить).

## Порядок реализации
1. Блок 3 (CI) — чистые файлы, без зависимости от стека.
2. Блок 1 (морфология) — код + переиндексация + проверка на живом стеке.
3. Блок 2 (k6) — прогон по живому стеку, фиксация чисел.

Блоки 1 и 2 требуют поднятого стека. Коммиты — от аккаунта пользователя, без атрибуции; `.env` не трогать.
