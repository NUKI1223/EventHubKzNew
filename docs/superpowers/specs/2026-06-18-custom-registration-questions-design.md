# Кастомные вопросы при регистрации на мероприятие — дизайн

**Дата:** 2026-06-18
**Статус:** утверждён (брейншторм), готов к плану реализации
**Подход:** A — JSON-хранение вопросов и ответов + серверная валидация в registration-service

## 1. Цель и контекст

Сейчас регистрация на NATIVE-мероприятие — один клик (`RegisterButton` → `POST /api/registrations/event/{id}`, без тела). Организатор не может собрать с участников дополнительные данные (размер футболки, уровень, откуда узнали и т.п.).

Фича добавляет организатору возможность задать **кастомные вопросы**, на которые участник отвечает при регистрации, а организатор видит ответы в кабинете и в Excel-выгрузке.

Решения, принятые на брейншторме:
- Вопросы задаются **в форме заявки** (`EventRequestForm`), **до одобрения**, и переносятся в `Event` при одобрении админом. Редактирование после одобрения — вне scope (см. §8).
- Типы вопросов: **свободный текст** и **одиночный выбор** (single). У каждого вопроса флаг «обязательный».
- Организатор смотрит ответы через **раскрывающуюся строку** в таблице участников + **колонки в Excel**.
- Вопросы имеют смысл только для NATIVE-регистрации.

## 2. Модель данных

### Общий тип в `eventhubkz-common`
Шарится event-service, registration-service и `EventDTO`.

```java
public enum QuestionType { TEXT, SINGLE }

public record QuestionDef(
    String id,            // стабильный id ("q1", "q2", …), генерится при добавлении в форме
    String label,         // текст вопроса
    QuestionType type,    // TEXT | SINGLE
    boolean required,
    List<String> options  // только для SINGLE; для TEXT — пусто/null
) {}
```

### Хранение
JSON-колонки через Hibernate-нативный `@JdbcTypeCode(SqlTypes.JSON)` → `jsonb`:

- `EventRequest.questions : List<QuestionDef>` — задаётся в форме заявки.
- `Event.questions : List<QuestionDef>` — копируется из заявки при одобрении.
- `EventRegistration.answers : Map<String,String>` — `{questionId → значение}`.
- `EventDTO.questions : List<QuestionDef>` — чтобы фронт отрисовал форму.

**Fallback:** если `jsonb` через `@JdbcTypeCode` даст ddl-проблемы — хранить как `TEXT` с ручной Jackson-сериализацией в сервисном слое. Поведение идентично.

**Обратная совместимость:** у существующих мероприятий/записей значения = `null` → трактуются как пусто. Миграция данных не нужна.

## 3. Поток данных

1. Заявка: организатор добавляет вопросы → `EventRequest.questions`.
2. Админ одобряет → `EventRequestService.approve()` переносит `questions` в новый `Event`.
3. Фронт открывает мероприятие → `EventDTO.questions` → рисует форму при «Записаться» (если список непустой).
4. Регистрация: `POST /api/registrations/event/{id}` получает тело `{ "answers": { "q1": "M", … } }` (раньше тело не использовалось).
5. **Валидация на сервере регистрации** (`EventRegistrationService.register`): берёт `event.questions` через существующий `EventClient.getEvent`, проверяет ответы (см. §5), сохраняет `answers`.

## 4. Авторизация ответов

Ответы видит только **организатор события или админ**.

- Публичный `GET /api/registrations/event/{id}` остаётся на минимальном `RegistrationView` **без** ответов (и без `code`).
- Эндпоинт, отдающий ответы, делает собственную проверку «организатор этого события или админ» через `EventClient` — тем же приёмом, что уже используется в `checkIn`.
- Агрегация `GET /api/events/{id}/attendees` (event-service, уже organizer-protected) обогащается: `AttendeeDTO` получает поле `answers`. event-service форвардит идентичность запросившего (`X-User-Id`/`X-User-Role`) в registration-service, чтобы ответы вернулись только авторизованному организатору.

Побочный эффект: это закрывает остаточную дырку C1 для чувствительных данных (ответы не утекают прямым запросом к registration-service).

## 5. Правила валидации (сервер)

`EventRegistrationService.register`:
- Загрузить `event.questions` через `EventClient`; построить карту `id → QuestionDef`.
- Каждый `required` вопрос → в `answers` есть непустое значение; иначе `400`.
- `SINGLE` → если отвечено, значение ∈ `options`; иначе `400`.
- Ключи ответов не из набора вопросов — отбрасываются.
- Кап длины текстового ответа — 1000 символов.
- Ошибка → `400` с понятным сообщением (`server.error.include-message: always` уже включён).

Лимиты при создании заявки (Bean Validation + фронт): ≤ 10 вопросов, ≤ 10 опций на single, `label` ≤ 200 символов.

## 6. Фронтенд

### 6.1 Редактор вопросов в `EventRequestForm`
Блок «Вопросы к участникам (необязательно)»: кнопка «+ Добавить вопрос»; для каждого — поле `label`, переключатель типа (Текст / Один из списка), тумблер «Обязательный», для single — редактор опций (добавить/удалить строки). Массив уходит в payload заявки как `questions`. Пустой список = вопросов нет (текущее поведение).

### 6.2 Модалка регистрации (`RegistrationModal`)
- Если `questions` непустой и юзер ещё не записан → клик «Записаться» (в `full` и `compact` вариантах) открывает модалку с формой: TEXT → `input/textarea`, SINGLE → `radio`/`select`. Клиентская проверка обязательных, затем `POST` с телом `{answers}`.
- Если вопросов нет → один клик, как сейчас.
- Отмена записи → простой `DELETE`, без формы.
- **Интеграция с `useToggleResource`:** добавить необязательный async-хук `onBeforeActivate()`, возвращающий тело запроса (или `null` = отмена). `RegisterButton` при наличии вопросов передаёт колбэк, открывающий модалку и резолвящийся ответами. Оптимистичное обновление / откат / тосты сохраняются. Без вопросов хук работает как сейчас.

### 6.3 Просмотр ответов организатором (`EventRegistrants`)
- У каждой строки участника — кнопка-раскрывашка «Ответы» (`вопрос → ответ`), данные из обогащённого `/attendees`.
- Excel-выгрузка: к колонкам (№/ID/Имя/Email/Статус) добавляется по колонке на каждый вопрос (заголовок = `label`).

### 6.4 Просмотр вопросов админом (`AdminEventRequests`)
При модерации заявки админ видит заданные вопросы read-only списком.

## 7. Граничные случаи
- Вопросы фиксируются в момент заявки и не редактируются после одобрения (MVP) → проблемы «вопрос изменили, а люди уже ответили» нет by design.
- Повторная регистрация идемпотентна: если уже записан — возвращаем существующую запись, ответы **не переписываем**. Сменить ответы → отменить и записаться заново.
- `id` вопроса стабилен и едет через одобрение; на него ссылаются ответы.
- Вопросы только у NATIVE; не-native до `register` не доходит.

## 8. Вне scope (на будущее)
- Редактирование вопросов в кабинете организатора после одобрения.
- Правка ответов после отправки.
- Множественный выбор (checkbox).
- Условные/ветвящиеся вопросы.
- Ответы на билете пользователя.
- Аналитика по ответам.

## 9. Тестирование
- **E2E** через gateway (Python `requests`): заявка-с-вопросами → одобрение админом → `register` без обязательного (`400`) → `register` с неверным значением single (`400`) → валидный `register` (`201`, ответы сохранены) → организатор видит ответы в `/attendees`, посторонний — нет/403.
- **Юнит** на логику валидации в `EventRegistrationService`.
- **Фронт:** сборка (`vite build`) + визуальный прогон (редактор вопросов, модалка регистрации, раскрывашка ответов, Excel) через playwright-поток (system Chrome), как в предыдущих проверках.

## 10. Затронутые компоненты (сводка)
- **eventhubkz-common:** `QuestionDef`, `QuestionType`; `EventDTO.questions`; тело запроса регистрации с `answers`.
- **event-service:** `EventRequest.questions`, `Event.questions`, перенос в `approve()`, `toDTO` → `questions`, `AttendeeDTO.answers`, форвард идентичности в `getAttendees`, лимиты в валидации заявки.
- **registration-service:** `EventRegistration.answers`, приём тела `{answers}` в `register`, серверная валидация через `EventClient`, organizer-only выдача ответов (проверка как в `checkIn`).
- **api-gateway:** маршруты не меняются (используются существующие пути).
- **frontend:** `EventRequestForm` (редактор), `RegistrationModal` (новый), `RegisterButton` + `useToggleResource` (`onBeforeActivate`), `EventRegistrants` (раскрывашка + Excel-колонки), `AdminEventRequests` (read-only вопросы).
