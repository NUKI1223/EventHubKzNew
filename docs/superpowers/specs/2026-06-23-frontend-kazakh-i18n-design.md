# Локализация фронтенда: казахский язык + переключатель RU/KK

**Дата:** 2026-06-23
**Контекст:** В отчёте казахский язык заявлен как ключевое преимущество (таблица 1.1), но в коде интерфейс только русский, i18n отсутствует. Эта спека описывает полную локализацию статического интерфейса на казахский с переключателем языка.

## Цель
Добавить переключатель языка RU/KK и перевести весь статический интерфейс сайта на казахский. Пользовательские данные (мероприятия, профили, теги) и строки, генерируемые бэкендом, не переводятся.

## Объём

**Переводим** (≈556 строк в ~42 `.jsx`-файлах, `frontend/src`): все статические UI-тексты — заголовки, кнопки, подписи полей, плейсхолдеры, навигация, статусы, сообщения интерфейса, пустые состояния, тексты лендинга.

**НЕ переводим:**
- данные из API: `event.title/description`, `user.username/description`, теги, и т.п.;
- строки, генерируемые бэкендом (тексты уведомлений вроде «Заявка одобрена», ответы AI-поддержки, серверные ошибки валидации) — они остаются русскими даже в режиме KK (вне объёма этой задачи; задокументировать как ограничение);
- комментарии в коде.

## Технологии
- **react-i18next** + **i18next** (индустриальный стандарт: ключи, фолбэк, интерполяция, смена языка на лету). Без `i18next-browser-languagedetector` — стартовый язык читаем сами из localStorage.

## Архитектура

- `frontend/src/i18n/index.js` — инициализация i18next:
  - `resources: { ru, kk }`;
  - `fallbackLng: 'ru'` (непереведённый ключ показывает русский, а не пустоту);
  - `lng` = `localStorage.getItem('lang')` либо `'ru'`;
  - `interpolation.escapeValue: false` (React сам экранирует).
  - Импортируется в `frontend/src/main.jsx` ДО рендера `App` (побочный импорт `import './i18n'`).
- `frontend/src/i18n/locales/ru.json` и `frontend/src/i18n/locales/kk.json` — плоско-вложенные словари, сгруппированные по экранам: `common.*`, `header.*`, `landing.*`, `events.*`, `eventDetail.*`, `profile.*`, `auth.*`, `requestForm.*`, `admin.*`, `notifications.*`, `support.*` и т.д. Ключи в обоих файлах идентичны по структуре.
- `frontend/src/components/LanguageSwitcher.jsx` — кнопки **RU / KK** в шапке (`Header.jsx`):
  - вызывает `i18n.changeLanguage(lng)` и `localStorage.setItem('lang', lng)`;
  - активный язык визуально подсвечен;
  - смена применяется на лету, без перезагрузки.

## Поток данных
Компонент → `const { t } = useTranslation()` → `t('namespace.key')` → строка из активного словаря; при отсутствии ключа в `kk` → фолбэк на `ru`. Интерполяция значений через `t('key', { name })` там, где в строку подставляются данные.

## Перевод
`ru.json` собирается из текущих хардкод-строк (фактически рефакторинг: строка из JSX → ключ + значение в ru.json). `kk.json` генерируется как перевод ru.json; **финальную вычитку и правку казахского делает пользователь** (фолбэк на ru страхует от пустых мест на время вычитки).

## Проверка (критерии приёмки)
- В шапке есть переключатель RU/KK; клик меняет язык всего статического каркаса на лету.
- Выбор языка переживает перезагрузку страницы (localStorage).
- Непереведённый ключ показывает русский текст, а не пустую строку / сырой ключ.
- Пользовательские данные (название события, имя профиля) остаются на языке, на котором их ввели, в обоих режимах.
- `npm run build` проходит.

## Декомпозиция реализации
1. Инфраструктура: установка зависимостей, `i18n/index.js`, пустые `ru.json`/`kk.json`, импорт в `main.jsx`, `LanguageSwitcher` в шапке + перевод `Header`/`common`. Уже даёт рабочий переключатель.
2. Перевод экранов группами (каждая группа — отдельная задача с самостоятельной проверкой): лендинг → список+карточка событий → деталь события → профиль → форма заявки/организатор → auth → admin → поддержка/уведомления (фронтовые) → прочее.

## Ограничения
- Бэкендовые строки остаются русскими в режиме KK (см. «НЕ переводим»).
- Качество казахского перевода — на финальной вычитке пользователя.
